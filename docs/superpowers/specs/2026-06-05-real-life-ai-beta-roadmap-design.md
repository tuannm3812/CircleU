# Real-Life AI Beta Roadmap Design

## Goal

Make Circleu useful as a real daily AI reflection app on an iPhone, not just a polished prototype.

The beta should let a user capture a real thought, receive a helpful AI reflection, save it into a journal, practice one small next action, and review patterns over time. The app should remain local-first for now, with clean boundaries for Apple Intelligence first and cloud AI/backend providers later.

## Product Principle

Build workflow-first, not screen-first.

Every tab should earn its place by supporting the user's reflection loop:

1. Capture what happened or what the user is feeling.
2. Analyze it with a private AI engine.
3. Save and edit the reflection.
4. Practice or act on one recommendation.
5. Review progress, patterns, and trusted sharing options.

The beta is successful when Tuan can use the app on a real iPhone for several days and each major flow feels connected, reliable, and worth keeping.

## Target User Outcomes

A beta user can:

- complete onboarding and understand what Circleu does,
- record or type a reflection when voice is unavailable,
- get an AI-generated summary, emotion, insight, and suggested practice,
- retry or regenerate analysis when the result is weak,
- save the result into Journal,
- edit title, emotion, notes, and tags later,
- start a practice from an AI suggestion,
- see daily progress, streaks, and emotional patterns,
- share only safe, intentional summaries,
- keep using the app without a backend account.

A developer or business teammate can:

- understand the app structure by feature, model, store, service, and component folders,
- test AI behavior on a phone through QA tools,
- inspect persisted local data,
- add backend sync later without rewriting product flows,
- compare Apple Intelligence, local fallback, and future model providers behind stable interfaces.

## Scope

This roadmap covers the next beta phase after the current local AI/session workspace work.

In scope:

- daily reflection flow refinement,
- onboarding completion,
- connected tab navigation,
- real Journal, Practice, Circle, and Profile workflows,
- visual polish aligned with the Figma direction,
- local AI quality and QA tools,
- local persistence improvements,
- backend-ready interfaces and documentation,
- phone test workflow.

Out of scope for this beta phase:

- production authentication,
- cloud database sync,
- paid subscriptions,
- real multi-user circle collaboration,
- push notifications,
- App Store release automation,
- live network AI provider calls unless added behind a local/offline-safe adapter.

## App Workflow Design

### 1. Onboarding

Onboarding should prepare the user for the daily loop.

Required flow:

1. Welcome and value statement.
2. Privacy and local-first reassurance.
3. Reflection preference setup.
4. Microphone and speech recognition permission education.
5. Finish into Home.

Onboarding should store completion locally and avoid reappearing unless reset from Profile QA tools.

### 2. Home

Home becomes the daily command center.

It should show:

- today's reflection status,
- primary record/type action,
- latest saved reflection,
- suggested practice from the latest AI result,
- lightweight progress snapshot,
- one clear next action.

Home should never feel like a static dashboard. It should guide the user into the next useful step.

### 3. Record Or Type Reflection

The capture flow should support real life conditions.

Required states:

- microphone permission missing,
- speech recognition unavailable,
- ready to record,
- actively recording,
- paused or stopped,
- transcript available,
- transcript too short,
- analyzing,
- analysis failed,
- analysis succeeded,
- typed fallback.

The user should be able to continue even if voice capture fails.

### 4. AI Reflection Result

The result screen should be a decision point, not a dead end.

It should show:

- summary,
- detected emotion,
- deeper insight,
- suggested practice,
- confidence or quality signal,
- engine label,
- regenerate action,
- save to Journal,
- start Practice,
- safe share action when appropriate.

The screen should make it clear what was generated by AI and what the user edited.

### 5. Journal

Journal becomes the user's memory layer.

Required workflow:

- browse entries by recency,
- search title, emotion, tags, and notes,
- filter by emotion or tag,
- open detail,
- edit title, emotion, private note, tags, and transcript note,
- inspect linked AI session history,
- regenerate from a saved entry,
- export or share a privacy-safe version.

Journal should use display fields that reflect user edits, not only the original AI result.

### 6. Practice

Practice turns insight into action.

Required workflow:

- show today's recommended practice,
- start a short guided practice,
- mark complete,
- connect completion to the source reflection when available,
- show streak or weekly completion count,
- allow choosing from recent AI-suggested practices.

Practice should be simple enough to complete in one minute during beta testing.

### 7. Circle

Circle should be useful locally before real backend collaboration exists.

Required workflow:

- preview what would be shared,
- generate a safe reflection card,
- copy/share a summary without private notes,
- show local circle feed demo entries clearly as local examples,
- explain pending backend requirements through product-safe wording only where necessary.

Circle should not pretend real multi-user sync exists yet.

### 8. Profile And QA

Profile should support both user settings and beta testing.

Required workflow:

- user display identity,
- privacy controls,
- reset onboarding,
- export local data,
- AI Lab for session inspection,
- seeded QA scenarios,
- phone test checklist access or summary,
- future provider configuration placeholder hidden behind QA/developer affordance.

Profile is the right location for testing tools until the product is ready for external users.

## AI Engine Design

### Output Contract

The app should standardize AI output around a single reflection result contract:

- title,
- summary,
- emotion,
- insight,
- suggested practice,
- follow-up prompt,
- confidence,
- safety/privacy note when needed,
- engine metadata.

The UI should not depend on provider-specific response shapes.

### Provider Strategy

Beta order:

1. Local deterministic fallback for reliability.
2. Apple Intelligence or Apple-native APIs where available.
3. Future cloud model adapter through backend or direct provider abstraction.

Every provider should conform to the same analysis interface so the product can switch engines without changing screens.

### QA Strategy

AI QA should support:

- testing typed transcripts,
- replaying saved sessions,
- comparing regenerated attempts,
- exporting session reports,
- identifying weak outputs,
- tracking engine name and failure states.

This keeps AI quality observable during real phone use.

## Data And Architecture Design

The repo structure should continue moving toward:

- `Models` for product data contracts,
- `Stores` for local persistence and app state,
- `Services` for AI, recording, backend prep, export, and identity,
- `Features` for screen-level product areas,
- `Components` for reusable UI building blocks,
- `Theme` or design system files for colors, typography, and spacing,
- `docs` for product, QA, architecture, and implementation plans.

Preferred architecture rule:

Views should compose workflows. Stores should own state changes. Services should own external capabilities. Models should remain simple and portable.

## Visual Design Direction

The beta should align more closely with the Figma direction while keeping native SwiftUI ergonomics.

Design priorities:

- consistent navigation bars,
- reusable section headers and action rows,
- refined empty states,
- professional color assets,
- readable iPhone 17 Pro layout,
- consistent spacing and card treatment,
- clear primary action hierarchy,
- no placeholder-looking text,
- accessible contrast and Dynamic Type tolerance.

The visual system should feel calm, private, and focused rather than decorative.

## Beta Phases

### Phase 1: Daily Reflection Beta

Complete the core loop:

1. onboarding,
2. Home daily hub,
3. record/type reflection,
4. AI result,
5. save to Journal,
6. reopen/edit Journal detail,
7. start suggested Practice.

This is the first implementation target.

### Phase 2: Practice And Progress

Make insight become behavior:

- guided practice sessions,
- completion tracking,
- weekly trend summary,
- emotion and tag patterns,
- practice history,
- better Home recommendations.

### Phase 3: Circle And Sharing

Make sharing trusted and intentional:

- local circle preview,
- privacy-safe share cards,
- selected entry sharing,
- export/copy improvements,
- backend collaboration design notes.

### Phase 4: Backend Readiness

Prepare for production without blocking beta:

- auth boundary,
- sync boundary,
- analytics boundary,
- provider boundary,
- migration strategy,
- backend API sketch.

## First Implementation Slice

The next build should be `Daily Reflection Beta v1`.

Deliverables:

- complete onboarding flow,
- Home daily hub refinement,
- production-like record/type flow states,
- AI result screen with save, regenerate, and practice actions,
- Journal list/detail using edited display fields,
- Practice start from AI suggestion,
- shared navigation and reusable UI components where they reduce duplication,
- updated phone test checklist,
- focused verification on simulator or connected iPhone.

Acceptance criteria:

- A new user can complete onboarding and land on Home.
- A user can record or type a reflection and get an AI result.
- A user can save the result to Journal.
- A user can reopen and edit the saved entry.
- A user can start a practice from the reflection.
- No main tab contains placeholder-only content.
- Private notes never appear in normal share/copy flows.
- The app builds successfully for iPhone simulator and remains ready for physical phone testing.

## Risks

- Voice and speech APIs can fail differently on simulator and physical devices.
- Apple Intelligence availability may vary by device, OS, region, and API access.
- Too much visual polish before workflow completion could slow beta learning.
- Backend-like UI promises could confuse users if real sync is not implemented yet.
- Local persistence changes need careful migration to avoid breaking existing test data.

## Decision

Optimize the next major improvement cycle for a 3-4 week real-life AI beta.

Prioritize a complete daily reflection loop first, then expand practice, progress, circle sharing, and backend readiness in that order.
